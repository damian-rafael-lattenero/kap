You have three async calls. They should run in parallel. Your code doesn't show it.

```kotlin
@KapTypeSafe
data class Dashboard(val user: String, val cart: String, val promos: String)

// KAP — what you mean is what you write
val dashboard: Dashboard = kap(::Dashboard)
    .withUser { fetchUser() }       // ┐ all three start at t=0
    .withCart { fetchCart() }        // │ total time = max(30, 20, 10) = 30ms
    .withPromos { fetchPromos() }   // ┘ not 60ms sequential
    .executeGraph()
```

Each `.withX` is generated by KSP from the constructor parameter names. The IDE autocompletes the next parameter. Swap two? **Doesn't compile.**

Now add a dependency: phase 2 needs phase 1's results.

```kotlin
@KapTypeSafe
data class UserContext(val profile: String, val prefs: String, val tier: String)

@KapTypeSafe
data class PersonalizedDashboard(val recs: String, val promos: String, val trending: String)

// Fetch context first, then personalize — two phases, one chain
val dashboard = kap(::UserContext)
    .withProfile { fetchProfile(id) }         // ┐ phase 1: parallel
    .withPrefs { fetchPreferences(id) }       // │
    .withTier { fetchLoyaltyTier(id) }        // ┘
    .andThen { ctx ->                         // ── barrier: ctx available
        kap(::PersonalizedDashboard)
            .withRecs { fetchRecommendations(ctx.profile) }    // ┐ phase 2: parallel
            .withPromos { fetchPromotions(ctx.tier) }          // │ uses ctx
            .withTrending { fetchTrending(ctx.prefs) }         // ┘
    }
    .executeGraph()
```

<summary><b>Same thing with raw coroutines</b></summary>

```kotlin
val dashboard = coroutineScope {
```

## Three concepts. That's it.

| You write | What happens | Think of it as |
|---|---|---|
| `.withX { }` | Runs in parallel with everything else in the same phase | "and at the same time..." |
| `.thenX { }` | Waits for all above, then continues | "once that's done..." |
| `.andThen { ctx -> }` | Waits, passes the result, then continues | "using what we got..." |

## And it scales.

11 microservice calls. 5 phases. Dependencies between them. One flat chain:

```kotlin
@KapTypeSafe
data class CheckoutResult(
    val user: UserProfile, val cart: ShoppingCart,
    val promos: PromotionBundle, val inventory: InventorySnapshot,
    val stock: StockConfirmation,
    val shipping: ShippingQuote, val tax: TaxBreakdown, val discounts: DiscountSummary,
    val payment: PaymentAuth,
    val confirmation: OrderConfirmation, val email: EmailReceipt,
)

val checkout: CheckoutResult = kap(::CheckoutResult)
    .withUser { fetchUser() }               // ┐
    .withCart { fetchCart() }                // ├─ phase 1: parallel
    .withPromos { fetchPromos() }           // │
    .withInventory { fetchInventory() }     // ┘
    .thenStock { validateStock() }          // ── phase 2: barrier
    .withShipping { calcShipping() }        // ┐
    .withTax { calcTax() }                  // ├─ phase 3: parallel
    .withDiscounts { calcDiscounts() }      // ┘
    .thenPayment { reservePayment() }       // ── phase 4: barrier
    .withConfirmation { generateConfirmation() }  // ┐ phase 5: parallel
    .withEmail { sendEmail() }                    // ┘
    .executeGraph()
```

<details>
<summary><b>Same thing with raw coroutines (30 lines)</b></summary>

```kotlin
val checkout = coroutineScope {
    val dUser = async { fetchUser() }
    val dCart = async { fetchCart() }
    val dPromos = async { fetchPromos() }
    val dInventory = async { fetchInventory() }
    val user = dUser.await()
    val cart = dCart.await()
    val promos = dPromos.await()
    val inventory = dInventory.await()

    val stock = validateStock()

    val dShipping = async { calcShipping() }
    val dTax = async { calcTax() }
    val dDiscounts = async { calcDiscounts() }
    val shipping = dShipping.await()
    val tax = dTax.await()
    val discounts = dDiscounts.await()

    val payment = reservePayment()

    val dConfirmation = async { generateConfirmation() }
    val dEmail = async { sendEmail() }

    CheckoutResult(
        user, cart, promos, inventory, stock,
        shipping, tax, discounts, payment,
        dConfirmation.await(), dEmail.await()
    )
}
```

</details>

## Type-safe by default

Annotate your class with `@KapTypeSafe`. KSP generates named builders where each step only shows the next parameter. The compiler catches swapped arguments. The IDE guides you.

```kotlin
@KapTypeSafe
data class User(val firstName: String, val lastName: String, val age: Int)

kap(::User)
    .withFirstName { fetchFirstName() }   // IDE only shows this
    .withLastName { fetchLastName() }     // then this
    .withAge { fetchAge() }              // then this
    .executeGraph()
```

No annotation? **Doesn't compile.** No silent fallback to unsafe positional mode.

It works on functions too — KSP generates a marker object for disambiguation:

```kotlin
@KapTypeSafe
fun buildGreeting(name: String, age: Int): String = "Hello $name, you are $age"

// BuildGreeting is a generated object
kap(BuildGreeting)
    .withName { fetchName() }
    .withAge { fetchAge() }
    .executeGraph()
```

For **third-party classes** you can't annotate, use `@KapBridge`:

```kotlin
@file:KapBridge(ThirdPartyUser::class)

kap(::ThirdPartyUser)
    .withFirstName { fetchFirstName() }
    .withLastName { fetchLastName() }
    .withAge { fetchAge() }
    .executeGraph()
```

For **third-party functions**, wrap in a one-line annotated delegate:

```kotlin
@KapTypeSafe
fun buildDashboard(userName: String, cartSummary: String) =
    com.thirdparty.buildDashboard(userName, cartSummary)

kap(BuildDashboard)
    .withUserName { fetchUserName() }
    .withCartSummary { fetchCartSummary() }
    .executeGraph()
```

**The rule:** classes and bridges use `kap(::ClassName)`. Functions use `kap(MarkerObject)` where the marker is a KSP-generated object.
